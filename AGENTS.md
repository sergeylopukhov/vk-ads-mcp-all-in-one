# Правила работы AI-агента с VK Ads MCP

## Маршрутизация проекта

- Целевой кодовый агент: Codex.
- Вопросы, анализ и обсуждение не являются разрешением менять файлы или выполнять запись в рекламном кабинете. Нужен явный запрос на реализацию или изменение.
- Внутренняя документация в `docs/` локальная: не коммитить, не отправлять в GitHub и не включать в релизы без отдельного запроса.
- Перед изменением кода, конфигурации, тестов, скриптов или поведения сервера прочитай `docs/code_rules.md`.
- При задачах, затрагивающих архитектуру, API, запуск, безопасность, интеграции, тестирование или документацию, сначала прочитай `docs/architecture.md` и профильный документ из него.
- Режим поддержки документации: automatic durable maintenance. После завершённого изменения обновляй только тот внутренний документ, чьи долговечные факты действительно изменились; не веди в docs журнал обычных задач.

## Начало задачи

1. Сначала вызови `get_provider_context` и убедись, что выбран нужный локальный профиль и режим.
2. Если нужна возможность неизвестна, найди её через `search_tools`. Не придумывай имя инструмента, API endpoint, параметр или ID.
3. Перед работой с конкретным объектом прочитай его по точному ID. Для списков используй `limit` и `offset`, не загружай кабинет целиком без причины.

## Аналитика

1. Сначала получи статистику за нужный период и уровень сущности.
2. Для сравнения, ранжирования, аномалий и рекомендаций используй `analytics_*`.
3. Не называй вывод причинностью, если данные показывают только корреляцию.
4. В отчётах не раскрывай токены, client secret, PII, банковские данные и закрытые URL.

## Создание и изменение

1. По умолчанию сервер работает в `readonly`; не проси включить запись, если задачу можно выполнить чтением.
2. Перед любым write вызови подходящий preflight: `ad_plan_preflight`, `ad_group_preflight`, `banner_preflight`, `media_validate` или `creative_preflight`.
3. Для записи используй только `write_preview` либо её именованный alias. Покажи пользователю payload, риск и ожидаемое изменение.
4. Выполняй `write_execute` только после явного подтверждения пользователя именно для свежего preview. Никогда не подставляй фразу подтверждения сам.
5. После успешной записи перечитай созданный или изменённый объект и сообщи ID, итоговый статус и результат проверки.

## Защита кабинета

- Не меняй существующие рабочие кампании, группы, баннеры, аудитории, счётчики и бюджеты.
- Для тестов разрешены только изолированные объекты с префиксом `__MCP_TEST__`; после теста удаляй только объекты этого сценария.
- Не обходи allowlist, `VK_ADS_MODE`, `VK_ADS_UPLOAD_DIR`, PII opt-in и ограничения профиля.
- Не передавай произвольные URL, файлы, API host или raw payload в обход инструментов сервера.
- Если сервер сообщает, что контракт не проверен, объясни ограничение и не выдавай предположение за рабочую функцию.
